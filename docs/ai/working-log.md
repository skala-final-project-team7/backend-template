- **Admin Key 말소 흐름 RabbitMQ completion event 방식으로 갱신**: 2026-06-04 에 정리했던 BFF Virtual Thread watcher + `/ml/ingest/status/{jobId}` polling 방식은 이번 결정으로 대체. `/api/admin/ingest` 는 RabbitMQ 기반 비동기 수집 플로우로 문서화했다. BFF 는 관리자 수집 요청에서 auth-server 내부 API 로 Admin Key 를 activate 한 뒤 RabbitMQ ingest job 을 직접 발행하거나 Data Ingestion Pipeline 의 `/ml/ingest` 에 job 발행을 위임한다. Data Ingestion Worker 는 job consume 후 `adminUserId` 로 auth-server 내부 credential 조회 API 를 호출해 admin OAuth `accessToken` + `cloudId` 를 함께 조회하고, Confluence 호출 시 `Authorization: Bearer {admin accessToken}` + `Atl-Confluence-With-Admin-Key: true` 헤더를 사용한다. RabbitMQ job/completion payload 에는 `jobId`, `adminUserId`, `mode`, `status` 등 작업 식별/상태 정보만 포함하고 `accessToken`/`refreshToken`/`cloudId` 는 포함하지 않는 보안 원칙을 명시했다. 작업 완료/실패 시 Data Ingestion Pipeline 이 completion event 를 발행하고 BFF consumer 가 이를 consume해 auth-server `POST /internal/admin/key/deactivate` 를 호출한다. deactivate 대상은 OAuth token 이 아니라 Atlassian Admin Key 활성 상태이며, `jobId` 기준 idempotent 로 취급한다. 장애 정책 초안도 추가: BFF 재시작/consumer 장애 시 durable queue event 재처리, Admin Key 60분 TTL fallback, deactivate 최대 5회 재시도 후 DLQ 이동, DLQ 는 원인 조치 뒤 event 재발행 또는 auth-server deactivate 수동 호출로 복구. 변경 파일: `docs/api-spec.md` v2.5.0·§1-4·§2-2·§3, `backend/bff-server/current-plans.md` 4단계 placeholder·RabbitMQ completion event 체크리스트, `backend/auth-server/current-plans.md` 내부 credential 조회 API·Admin Key activate/deactivate 관계, `docs/adr/0001-page-level-acl-source.md` §2.1 역할 분리와 MQ payload 원칙.
